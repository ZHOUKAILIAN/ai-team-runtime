## 📋 这是什么？

一个让你**一个人当一个团队**的开发流程。你只需要：
1. 说清楚要做什么（用自然语言）
2. AI 团队自动帮你完成（产品、开发、测试、验收）
3. 你最后检查一下就能发布

**核心理念**：你是 CEO，AI 是你的产品经理+开发+测试团队。

## 🚀 实际使用流程

### 场景：你想做一个新功能

假设你想做一个"用户登录"功能。

#### 步骤 1️⃣：初始化项目文档（可选，首次推荐）

**你做什么**：
```bash
# 在 Claude Code 中
/ai-doc-driven-dev:init-doc-driven-dev
```

**发生了什么**：
- 自动创建标准文档目录结构（`docs/prd/`, `docs/architecture/` 等）
- 生成 PRD 模板

**什么时候跳过这步**：
- 项目已经有文档结构了
- 你想直接开始（也可以）

---

#### 步骤 2️⃣：说清楚需求和验收标准

**你做什么**（选一种方式）：

```bash
# 方式 A：直接说需求（最简单，但要说清楚验收标准）
执行这个需求：做一个用户登录功能，支持手机号+验证码，登录成功后跳转首页，失败要提示错误信息

# 方式 B：如果你已经写了 PRD 文档（推荐重要功能）
/company-run 根据 docs/prd/用户登录.md 构建功能

# 方式 C：指定平台（跳过后续选择）
执行这个需求：为小程序做一个用户登录功能，验证码60秒后可重发，登录后显示用户昵称
```

⚠️ **关键**：你的需求里要说清楚"怎么算成功"，比如：
- ✅ 好：登录成功后跳转首页，失败提示错误
- ❌ 差：做一个登录功能（没说怎么验收）

💡 **验收标准示例**：
```
执行这个需求：
做一个商品列表页
验收标准：
1. 每页显示20个商品
2. 支持上拉加载更多
3. 点击商品跳转详情页
4. 加载失败显示重试按钮
```

**发生了什么**（全自动，你不用管）：

```
📋 产品经理（AI）：
   - 把你的需求整理成详细的 PRD
   - 定义验收标准
   - 输出：产品规格文档

💻 开发工程师（AI）：
   - 读 PRD
   - 设计技术方案
   - 写代码
   - 输出：技术文档 + 代码实现

🧪 QA 工程师（AI）：
   - 问你：要在哪个平台测试？（小程序/网页/都要）
   - 自动跑测试（技术验收）
   - 发现问题就打回给开发修
   - 输出：测试报告

📦 运营经理（AI）：
   - 生成发布说明
   - 整理功能亮点

⚖️ 验收经理（AI）：
   - 汇总所有产物（PRD、代码、测试报告、发布说明）
   - 对照原始需求做业务验收
   - ⏸️ **停下来，等你（人类）做最终决策：Go / No-Go**
```

---

#### 步骤 3️⃣：选择测试平台（如果需要）

**AI 会问你**：
```
需要在哪些平台上验证？
A: 小程序
B: 网页
C: 两者都要
```

**你回答**：选 A/B/C

💡 **省事技巧**：在步骤 2 就说清楚"为小程序做..."，就不会问了。

---

#### 步骤 4️⃣：你来做最终验收（人工决策）

**AI 团队完成后会停下来，等你验收**

**命令**：
```bash
cd ~/.codex/vendor/ai-team
python3 -m ai_company review
```

**你要验收什么**（这是你的职责）：

1. **查看 AI 验收经理的建议**：
   ```bash
   cat .ai_company_state/artifacts/<session_id>/acceptance_report.md
   ```

   AI 验收经理会给出建议：
   - ✅ 建议通过（`accepted`）→ AI 觉得符合要求
   - ❌ 建议拒绝（`rejected`）→ AI 发现不符合需求

   **但最终决策权是你的！**

2. **检查所有产物**：
   ```bash
   .ai_company_state/artifacts/<session_id>/
   ├── prd.md              # 产品文档 - 看看 AI 理解对了吗？
   ├── tech_spec.md        # 技术方案 - 架构合理吗？
   ├── qa_report.md        # 测试报告 - 测试通过了吗？
   ├── release_notes.md    # 发布说明 - 功能描述对吗？
   └── acceptance_report.md # AI 验收建议 - AI 的判断
   ```

3. **根据验收标准做决策**：

   回顾你在步骤 2 定的验收标准，逐条检查：

   ```bash
   # 你的原始需求（步骤2）：
   执行这个需求：做用户登录，验证码60秒可重发，登录成功显示昵称

   # 现在检查：
   ✅ 能收到验证码吗？
   ✅ 60秒内按钮是灰色的吗？
   ✅ 登录后能看到昵称吗？
   ✅ 错误提示清晰吗？
   ```

4. **你的决策**：
   - **通过** → 继续步骤 5（发布）
   - **拒绝** → 看下面"如果验收不通过"

---

#### 步骤 5️⃣：发布或返工

**情况 A：你验收通过** ✅

```bash
# 代码已经写好了
# 测试已经通过了
# 文档已经生成了
👉 直接部署发布！
```

**情况 B：你验收不通过** ❌

先判断问题类型：

| 问题类型 | 表现 | 解决方案 |
|---------|------|----------|
| **需求理解错误** | AI 做的不是你想要的 | 回到步骤 2，重新说清楚需求+验收标准 |
| **验收标准不明确** | AI 不知道做到什么程度 | 补充验收标准，重跑步骤 2 |
| **技术实现有问题** | QA 测试通过但实际有 bug | 查看 `qa_report.md`，补充测试用例，重跑 |
| **缺少边界情况** | 正常场景 OK，异常场景没做 | 在需求中补充异常场景，重跑步骤 2 |

**返工示例**：

```bash
# 第一次（验收不通过）
执行这个需求：做一个登录功能
# AI 做完了，但你发现：没有错误提示！

# 第二次（补充验收标准）
执行这个需求：
做一个登录功能
验收标准：
1. 登录成功跳转首页
2. 密码错误提示"密码不正确"  ← 新增
3. 网络失败提示"网络连接失败" ← 新增
```

## 📝 如何写好需求和验收标准

### 为什么验收标准重要？

验收标准决定了"什么叫做完了"。如果你一开始不说清楚，AI 团队可能做出来的不是你想要的。

### 好的验收标准长什么样？

**✅ 好的例子**：
```bash
执行这个需求：
做一个用户登录功能

功能要求：
1. 支持手机号+验证码登录
2. 验证码60秒后可以重新发送
3. 输入手机号格式错误时提示"请输入正确的手机号"
4. 登录成功后跳转到首页，并显示用户昵称
5. 登录失败提示"验证码错误，请重试"

验收标准：
1. 能正常收到验证码短信
2. 正确的验证码能登录成功
3. 错误的验证码提示失败
4. 60秒内"发送验证码"按钮是灰色的
5. 登录后首页能看到用户昵称
```

**❌ 不好的例子**：
```bash
执行这个需求：做一个登录功能
# 问题：没说怎么登录、登录后去哪、失败怎么办
```

### 验收标准模板

```bash
执行这个需求：
做一个 [功能名称]

功能要求：
1. [核心功能1]
2. [核心功能2]
3. [边界情况处理]

验收标准：
1. [正常流程] - 用户能做什么
2. [异常流程] - 出错时怎么办
3. [边界情况] - 极端情况怎么处理
4. [性能要求] - 多快算合格（可选）
5. [UI/UX要求] - 显示什么内容（可选）
```

### 常见场景的验收标准

#### 场景1：列表页
```bash
执行这个需求：做一个商品列表页

功能要求：
1. 显示商品图片、名称、价格
2. 支持下拉刷新
3. 支持上拉加载更多
4. 点击商品进入详情页

验收标准：
1. 首次加载显示前20个商品
2. 下拉刷新能获取最新数据
3. 上拉加载更多，每次加载20个
4. 到底了显示"没有更多了"
5. 加载失败显示"加载失败，点击重试"按钮
6. 点击商品能正确跳转到详情页
```

#### 场景2：表单提交
```bash
执行这个需求：做一个用户注册表单

功能要求：
1. 收集：手机号、密码、确认密码、验证码
2. 提交前校验输入格式
3. 提交成功跳转登录页

验收标准：
1. 手机号必须是11位数字
2. 密码至少6位
3. 两次密码必须一致
4. 验证码60秒内不能重发
5. 所有字段都填写才能点击"注册"按钮
6. 提交成功提示"注册成功"并跳转登录页
7. 提交失败显示服务器返回的错误信息
```

#### 场景3：数据展示
```bash
执行这个需求：做一个订单详情页

功能要求：
1. 显示订单基本信息（订单号、状态、金额）
2. 显示商品列表
3. 显示物流信息（如果已发货）
4. 支持取消订单（待支付状态）

验收标准：
1. 页面能正确展示所有订单信息
2. 待支付订单显示"取消订单"按钮
3. 已发货订单显示物流公司和单号
4. 点击"取消订单"弹出确认框
5. 确认取消后订单状态变为"已取消"
6. 加载失败显示"加载失败，点击重试"
```

### 实用技巧

**技巧1：用"能不能"来写验收标准**
```bash
# ✅ 好
验收标准：
1. 能看到商品列表
2. 能下拉刷新
3. 点击商品能跳转详情

# ❌ 差
验收标准：
1. 商品列表
2. 刷新
3. 详情
```

**技巧2：写清楚异常情况**
```bash
验收标准：
1. 正常：登录成功跳转首页
2. 异常：密码错误提示"密码不正确"
3. 异常：网络失败提示"网络连接失败，请重试"
4. 边界：5次密码错误后锁定账号30分钟
```

**技巧3：第一次不用写太详细**
```bash
# 第一次运行
执行这个需求：做一个登录功能，支持手机号登录

# AI 团队做完后，你发现缺东西
# 第二次运行补充
执行这个需求：做一个登录功能，支持手机号登录，登录失败要提示错误，登录成功显示用户头像
```

## 🚨 常见问题

### Q1：AI 一直问我问题停不下来

**原因**：你的需求太模糊了

**解决**：
```bash
# ❌ 不好的需求
执行这个需求：做一个功能

# ✅ 好的需求
执行这个需求：做一个用户登录功能，支持手机号+验证码，登录后跳转首页
```

### Q2：验收一直被拒绝

**原因**：可能是实现不符合需求

**解决**：
```bash
# 1. 看拒绝原因
cat .ai_company_state/artifacts/<session_id>/acceptance_report.md

# 2. 如果是需求不清楚
执行这个需求：做登录功能（补充更多细节）...

# 3. 如果是实现问题
# AI 会自动修复，你重跑就行
执行这个需求：<相同需求>
```

### Q3：我想自己改代码，不想全让 AI 做

**没问题！**：
```bash
# 1. 让 AI 团队跑完流程（生成文档和初始代码）
执行这个需求：做登录功能

# 2. 拿到产物后，你自己修改代码
cd .ai_company_state/artifacts/<session_id>/
# 查看 tech_spec.md，理解设计
# 自己写代码

# 3. 手动测试
# 不需要重跑 AI 团队
```

### Q4：找不到我的产物文件

**位置**：
```bash
# 所有产物都在这里
~/.codex/vendor/ai-team/.ai_company_state/

# 最新一次的 session_id 在 review 命令输出中
cd ~/.codex/vendor/ai-team
python3 -m ai_company review  # 会显示 session_id

# 然后去这个目录
cd .ai_company_state/artifacts/<session_id>/
```

### Q5：我想在 CI/CD 里用

```bash
# 在你的 CI 脚本中
cd ~/.codex/vendor/ai-team

  --request "$(cat docs/prd/feature.md)" \
```

## 📈 使用建议

### ✅ 推荐做法

1. **需求说清楚**
   - 好：做一个用户登录功能，手机号+验证码，登录后跳转首页
   - 差：做个登录

2. **第一次用先简单试试**
   ```bash
   执行这个需求：写一个 Hello World 页面
   ```
   熟悉流程后再做复杂功能

3. **重要功能先写 PRD**
   ```bash
   /ai-doc-driven-dev:init-doc-driven-dev
   # 填写 docs/prd/功能名.md
   /company-run 根据 docs/prd/功能名.md 构建
   ```

4. **查看学习记忆**
   定期看看 `.ai_company_state/memory/` 里学到了什么

5. **复杂功能分步骤**
   ```bash
   # 不要：做一个完整的电商系统
   # 要：
   执行这个需求：做商品列表页
   # 等完成...
   执行这个需求：做商品详情页
   # 等完成...
   执行这个需求：做购物车
   ```

### ❌ 不推荐做法

1. **需求模糊就开始**
   - 会导致反复返工

2. **跳过验收直接用**
   - 可能有隐藏问题

3. **忽略拒绝原因**
   - 说明真的有问题

4. **同时跑多个 session**
   - 等一个完成再开始下一个

5. **手动删除学习记忆**
   - 除非确定记错了

## 🎯 总结：你的实际工作流

```
1. 你：说清楚需求和验收标准
   执行这个需求：做一个 xxx 功能，验收标准是...
         ↓
2. AI 团队：自动跑完（你不用管）
   📋 产品经理：整理 PRD + 定义验收标准
   💻 开发工程师：写代码
   🧪 QA 工程师：跑测试（技术验收）
   📦 运营经理：写发布说明
   ⚖️ 验收经理：做业务验收，给出建议
         ↓
3. AI 验收经理：停下来，给你建议
   "我建议通过/拒绝，原因是..."
         ↓
4. 你：做最终决策（人工验收）
   - 看所有产物（PRD、代码、测试报告）
   - 对照验收标准逐条检查
   - 决策：
     ✅ 通过 → 发布
     ❌ 不通过 → 补充需求，回到步骤 1
```

**关键点**：
- ✅ AI 团队会做技术验收（QA 测试）
- ✅ AI 验收经理会做业务验收（给建议）
- ⚠️ **但最终决策权是你的**（人工验收）
- 💡 验收标准在步骤 1 就要定好
